# Data Interface — LLM Processing Pipeline

This document defines the **data exchange format** between the **News Crawler Team** (data providers) and the **LLM Processing Pipeline** (content summarization and transformation).

It ensures both systems can exchange news articles seamlessly via JSON or API-based integration.

---

## 1. Overview

The crawler team will deliver structured news article data in a standardized JSON format.  
This serves as the **input** to the LLM Processing Pipeline.

The LLM pipeline will then output processed data (summaries, style-transformed text, etc.) in a similar JSON structure, adding its own results and metadata.

---

## 2. Expected Input Format (Crawler to LLM Pipeline)

Each article must be represented as a JSON object with the following fields:

| Field | Type | Description |
|-------|------|--------------|
| `id` | string | Unique identifier for the article. |
| `source` | string | News source name (e.g., “BBC”, “Reuters”). |
| `url` | string | Original article URL. |
| `title` | string | Headline text. |
| `body` | string | Full raw text of the article. |
| `language` | string | Two-letter code (“en”, “de”). |
| `published_at` | string (ISO 8601) | Publication timestamp. |
| `retrieved_at` | string (ISO 8601) | When the crawler fetched it. |
| `category` | string | Optional (e.g., “politics”, “sports”). |
| `author` | string | Optional. |
| `meta` | object | Optional metadata such as sentiment, tags, word count. |


### Example

```json
{
  "id": "BBC-20251031-001",
  "source": "BBC",
  "url": "https://bbc.com/news/economy",
  "title": "UK economy grows faster than expected",
  "body": "The UK economy expanded by 0.5% in the third quarter...",
  "language": "en",
  "published_at": "2025-10-31T07:00:00Z",
  "retrieved_at": "2025-10-31T07:05:00Z",
  "meta": { "word_count": 420 }
}
```

## 3. NEW! LLM Pipeline Batch Input Format for now summarizing articles in batches
**Top-Level Fields**
| Field        | Type             | Required | Description                                                                                                      |
| ------------ | ---------------- | ---------| ---------------------------------------------------------------------------------------------------------------- |
| `request_id` | string           | Yes      | Unique ID for the summarization request (generated by n8n). Used for tracing, logging, and debugging. |
| `filter`     | object           | Yes      | Contains filtering parameters used to select or group articles (e.g., time window, category, keywords).          |
| `options`    | object           | Optional | Contains user-defined settings such as summary style and sentiment tone.                                         |
| `articles`   | array of objects | Yes      | List of articles to be processed and summarized as a batch. Must contain at least 1 article.                     |


**`filter` Object Fields
| Field         | Type             | Required | Description                                                                                                   |
| ------------- | ---------------- | ---------| ------------------------------------------------------------------------------------------------------------- |
| `time_window` | string           | Optional | Time-based filter applied to the article selection (e.g., `"last_24_hours"`, `"last_5_minutes"`).             |
| `category`    | string           | Optional | Category filter (e.g., `"politics"`, `"sports"`, `"economy"`).                                                |
| `keywords`    | array of strings | Optional | Keyword filtering applied before summarization. Useful for targeted summaries (e.g., `["election", "vote"]`). |

**`options` Object Fields (to control tone and style of output)
| Field              | Type   | Required  | Description                                                                                                  |
| ------------------ | ------ | ----------| ------------------------------------------------------------------------------------------------------------ |
| `style_version`    | string |  Optional | Preferred summary tone or style. Examples: `"neutral"`, `"journalistic"`, `"short"`, `"formal"`, `"simple"`. |
| `sentiment_target` | string |  Optional | Optional sentiment framing. `"none"` means no bias. Other values: `"positive"`, `"negative"`, `"balanced"`.  |

**`articles` Array (Article Object Fields)
| Field          | Type              | Required| Description                                                                                                         |
| -------------- | ----------------- | --------| ------------------------------------------------------------------------------------------------------------------- |
| `id`           | string            |  Yes    | Unique identifier for each article (created by crawler). Used to reference source IDs in chunk summaries.           |
| `title`        | string            |  Yes    | Original article headline (not used for summarization but useful for reference, logging, and potential UI display). |
| `body`         | string            |  Yes    | Main content of the article. This is the text that goes through cleaning, chunking, and summarization.              |
| `language`     | string            |  Yes    | ISO code for article language (`"en"`, `"de"`). Used for translation or multilingual processing.                    |
| `source`       | string            |  Yes    | Source or publisher (e.g., BBC, CNN, Reuters).                                                                      |
| `published_at` | string (ISO 8601) |  Yes    | Timestamp when the article was published. Used for time-window filters like “last 24 hours.”                        |

### Example
```json
{
  "request_id": "summary_batch_2025-11-12T15:30Z",
  "filter": {
    "time_window": "last_24_hours",
    "category": "politics",
    "keywords": ["election", "vote"]
  },
  "options": {
    "style_version": "neutral", 
    "sentiment_target": "none"
  },
  "articles": [
    {
      "id": "BBC-2025-001",
      "title": "Article title",
      "body": "Full text...",
      "language": "en",
      "source": "BBC",
      "published_at": "2025-11-12T12:31Z"
    },
    {
      "id": "CNN-2025-023",
      "title": "Another article...",
      "body": "Full text...",
      "language": "en",
      "source": "CNN",
      "published_at": "2025-11-12T08:10Z"
    }
  ]
}
```

## 4. NEW! LLM Pipeline Output Format (LLM to Frontend)
**Top-Level Fields**
| Field             | Type              | Required  | Description                                                                                                                        |
| ----------------- | ----------------- | ----------| ---------------------------------------------------------------------------------------------------------------------------------- |
| `request_id`      | string            |  Yes      | Unique ID matching the input request. Used for tracing, logging, and debugging.                                                    |
| `summary_type`    | string            |  Yes      | Always `"batch_summary"` for this version. Helps frontend/backend distinguish from future summary types (e.g., cluster summaries). |
| `filters_used`    | object            |  Yes      | Echo of the filters applied during the request. Ensures transparency and reproducibility.                                          |
| `article_count`   | integer           |  Yes      | Number of articles included in the batch after filtering.                                                                          |
| `language`        | string            |  Yes      | Output summary language (`"en"` or `"de"`).                                                                                        |
| `model`           | string            |  Yes      | Model name used for summarization (e.g., `"bart-large-cnn"` or `"llama3"`).                                                        |
| `style_version`   | string            |  Optional | Final style applied to the summary. Examples: `"neutral"`, `"formal"`, `"journalistic"`, `"short"`.                                |
| `sentiment`       | string            |  Optional | Overall sentiment of the generated summary if sentiment control was applied (`"none"`, `"positive"`, `"negative"`, `"balanced"`).  |
| `final_summary`   | string            |  Yes      | The global summary combining all chunk summaries into one coherent output.                                                         |
| `chunk_summaries` | array             |  Yes      | List of summaries generated from each chunk after hierarchical summarization. Useful for debugging and evaluation.                 |
| `processed_at`    | string (ISO 8601) |  Yes      | Timestamp when summarization completed.                                                                                            |

`filers_used` Object ZFields
| Field         | Type             | Required  | Description                                               |
| ------------- | ---------------- | ----------| --------------------------------------------------------- |
| `time_window` | string           |  Optional | Time filter used (e.g., `"last_24_hours"`).               |
| `category`    | string           |  Optional | Category applied (politics, sports, tech, etc.).          |
| `keywords`    | array of strings |  Optional | Keywords applied to filter articles before summarization. |

`chun_summaries` Array Fields
| Field                | Type             | Required  | Description                                                                        |
| -------------------- | ---------------- | --------- | ---------------------------------------------------------------------------------- |
| `chunk_id`           | integer          |  Yes      | Sequential index of the chunk.                                                     |
| `summary`            | string           |  Yes      | Summary of that specific chunk of text.                                            |
| `source_article_ids` | array of strings |  Optional | IDs of articles that contributed text to this chunk (traceability & transparency). |

#### Example
```json
{
  "request_id": "summary_batch_2025-11-12T15:30Z",
  "summary_type": "batch_summary",
  "filters_used": {
    "time_window": "last_24_hours",
    "category": "politics",
    "keywords": ["election", "vote"]
  },
  "article_count": 12,
  "language": "en",
  "model": "bart-large-cnn",
  "style_version": "neutral",
  "sentiment": "none",
  "final_summary": "Over the last 24 hours, political news has focused on...",
  "chunk_summaries": [
    {
      "chunk_id": 1,
      "summary": "The election commission released new data...",
      "source_article_ids": ["BBC-2025-001", "CNN-2025-023"]
    }
  ],
  "processed_at": "2025-11-12T15:31Z"
}
```

## Data Transfer Mechanism / Integration

To enable parallel development and ensure smooth future integration between the News Crawler and the LLM Processing Pipeline, the following staged approach to data transfer is proposed.

**Phase 1 – Development:**
The LLM team will use local JSON file exchange (/data/input/ and /data/output/ directories) for testing and validating the pipeline independently of the crawler’s progress.

**Phase 2 – Integration:**
Once the crawler module is ready, data will be sent via a REST API call to the FastAPI endpoint (/summarize).
This allows direct, real-time exchange of article data and ensures compatibility between both systems.

**Phase 3 – Full System Orchestration:**
The final integrated version will be coordinated through n8n, where both the crawler and LLM modules will run as part of a shared workflow.
This setup provides automated scheduling, execution logging, and traceability across the entire system.

This phased strategy allows each team to work in parallel, minimizes dependency delays, and results in a flexible, scalable integration path.

#### PREVIOUS! Evaluation Output Format (Evaluation Module to Project Leads / QA Reports)

After summarization, the Evaluation module analyzes each generated summary against reference summaries.
It outputs a structured JSON file containing quantitative quality metrics and optional qualitative assessments.

| Field | Type | Description |
|-------|------|--------------|
| `id` | string | Matches input article ID or summary ID|
| `rouge_1` | float | ROUGE-1 score. Measures unigram (word-level) overlap between model summary and reference. |
| `rouge_l` | float | ROUGE-L score. Measures longest common subsequence overlap, capturing fluency and structure.|
| `bert_f1` | float | BERTScore F1. Captures semantic similarity using transformer embeddings.|
| `bleu` | float (optional for bilingual runs) | BLEU score. Precision-based metric, useful for multilingual or translation evaluation.|
| `language` | string | Language of the evaluation sample (e.g., “en” or “de”). |
| `evaluated_at` | string (ISO 8601) | Timestamp when the evaluation metrics were computed.|

##### Example

```json
{
  "id": "BBC-20251031-001",
  "rouge_1": 0.46,
  "rouge_l": 0.39,
  "bert_f1": 0.87,
  "bleu": 0.42,
  "evaluated_at": "2025-11-12T11:40:00Z"
}
```

# LLM Summarization API – Input & Output Contracts

This document defines the **input and output schemas** for all LLM summarization endpoints.

---

## Common Concepts (Applies to All Endpoints)

### Article Object (Input)

```json
{
  "id": "string",
  "title": "string",
  "body": "string",
  "language": "en",
  "published_at": "ISO-8601 string (optional)",
  "source": "string (optional)",
  "category": "string (optional, upstream metadata)"
}
```

**Notes**

* Articles are assumed to be **pre-filtered upstream**
* `category` is **not used** by the LLM pipeline for decision-making
* Language is expected to be `"en"` for now (translation for M3)

---

### Metadata Fields (Common)

```json
{
  "request_id": "string",
  "filters_used": { "any": "json" },
  "processed_at": "ISO-8601 timestamp"
}
```

### filters_used: { "any": "json" }

A free-form object describing how the articles were filtered before they reached the LLM pipeline.

The backend does not apply filters, but records them for transparency and debugging.

{
  "category": "Politics",
  "date_range": "last_24_hours",
  "keywords": ["Ukraine", "NATO"],
  "language": "en" (translation now happens on the FE so this would not be sent to the LLM pipeline)
}

# 2. `/cluster_summary`

### Purpose

Cluster articles by topic, generate **one summary per cluster**, and assign:

* a **subcategory (cluster topic)**
* a **fixed top-level category (1 of 5 hard categories)**

---

### Output Categories

```text
Global Politics
Economics
Sports
Events
General News
```

---

### Input

```json
{
  "request_id": "string",
  "filters_used": {},
  "articles": [Article]
}
```

---

### Output

```json
{
  "request_id": "string",
  "summary_type": "cluster_summary",
  "cluster_count": 5,
  "clusters": [
    {
      "cluster_id": 0,
      "category": "Global Politics",
      "subcategory": "Ukraine War Developments",
      "topic_summary": "string",
      "articles": [
        { "url": "string", "title": "string" }
      ]
      "article_count": 2,
      "keywords": {
        "lda": [
          "ukraine war",
          "russian attack",
          "kyiv airstrike"
        ],
        "tfidf": [
          "zelenskyy",
          "missile strike",
          "air defense"
    ]
    }
  ],
  "processed_at": "ISO-8601 timestamp"
}
```

**Notes**

* `subcategory` is dynamically generated by LLaMA
* Noise clusters are always mapped to `General News`

# Differences bewteen `/cluster_summary` and `/summarize_clustered`
## /cluster_summary
* No mega summary — only per-cluster summaries
* Includes keywords — LDA and TF-IDF keywords per cluster
* Full article data — returns complete Article objects
* Sequential processing — slower (one cluster at a time)
* No request limit

## /summarize_clustered
* Includes mega summary — global summary combining all clusters
* No keywords — no LDA/TF-IDF extraction
* Lightweight articles — only URL + title (not full objects)
* Parallel processing — faster (3-5x, processes clusters simultaneously)
* Request limit — max 100 articles

## Quick decision guide
Use `/cluster_summary` if:
* You need keyword extraction
* You need full article data
* You're doing detailed analysis

Use `/summarize_clustered` if:
* You need a global mega summary
* You want faster processing
* You only need article references

# 4. `/summarize_with_style`

### Purpose

Generate a single summary and rewrite it using:

* **Writing style**
* **Output format**
* **Editorial tones (this is excluded for now until M3 based on decisions in class but you can add it if you just want a placeholder)**

(No clustering, no categories)

---

### Allowed Writing Styles

```text
journalistic
academic
executive
```

### Allowed Output Formats

```text
paragraphs
bullet_points
sections
tldr
```

---

### Input

```json
{
  "articles": [Article],
  "writing_style": "journalistic",
  "output_format": "sections"
}
```

---

### Output

```json
{
  "initial_summary": "string",
  "rewritten_summary": "string",
  "writing_style": "journalistic",
  "output_format": "sections"
}
```

**Notes**

* Tone (liberal / conservative / neutral) is intentionally excluded for now (until M3)
* Format may be auto-normalized internally if incompatible

---

## Design Guarantees

* Input articles are **never mutated**
* All outputs are **deterministic in structure**
* Categories are **fixed and controlled**
* Subcategories are **semantic, human-readable topics**
* No endpoint depends on frontend category metadata
